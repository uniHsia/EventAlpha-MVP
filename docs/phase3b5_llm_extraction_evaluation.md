# Phase 3B.5: LLM Extraction Evaluation and Normalization

Phase 3B.5 does not replace the causal reasoning agent. It first evaluates and hardens LLM event extraction, because weak extraction will contaminate every downstream step: credibility scoring, causal chains, asset mapping, ledger entries, and review.

## Why Evaluate Before Causal Reasoning

Event extraction is the system boundary where free-form news becomes structured data. Real LLM output can identify the correct `event_type`, but may also invent audit IDs, overfill timestamps, use non-standard asset names, or omit key entities. These issues must be fixed before asking an LLM to produce causal chains.

## Normalization

`eventalpha/rules/asset_aliases.yaml` maps common asset aliases to standard EventAlpha asset names. `AssetNormalizationService` applies exact, alias, whitespace-insensitive, and case-insensitive matching.

Unknown assets are preserved and reported as warnings. They are not silently deleted, because unknown names may contain useful information for later mapping expansion.

## System-Controlled Fields

`event_id`, `raw_id`, and `created_at` are internal audit fields. The LLM is not allowed to control them:

- `raw_id` is copied from the input `RawNews`.
- `event_id` is regenerated by the system.
- `created_at` is regenerated by the system.

This keeps the ledger auditable even when a real model fabricates plausible-looking IDs.

## Suspicious Event Time

If the LLM returns an `event_time` that is not explicitly present in the raw title or text, the post-processor clears it and records a warning. Supported explicit date forms include:

- `YYYY-MM-DD`
- `YYYY/MM/DD`
- `YYYY年M月D日`
- `M月D日`

This prevents a model from treating the current runtime date as the event occurrence time.

## Evaluation Script

Offline mock evaluation:

```bash
python scripts/evaluate_llm_extraction.py
```

Real OpenAI-compatible evaluation:

```bash
python scripts/evaluate_llm_extraction.py --real-llm --write-report
```

Evaluate one case:

```bash
python scripts/evaluate_llm_extraction.py --case 0
```

The report can be written to:

```text
reports/llm_extraction_eval.json
reports/llm_extraction_eval.md
```

## Report Fields

The evaluation summary includes:

- `total_cases`
- `event_type_match_count`
- `status_match_count`
- `asset_hint_overlap_avg`
- `entity_overlap_avg`
- `llm_unknown_asset_count`
- `event_time_suspicious_count`
- `postprocess_warning_count`
- `failed_case_count`

This is an engineering quality check, not a statistical benchmark.

## DeepSeek and Compatible APIs

Some OpenAI-compatible providers may not support the same structured-output response format as OpenAI. EventAlpha already falls back to JSON-object or prompt-only JSON mode, but the output is still always validated with Pydantic and then post-processed.

Common real-model issues to watch:

- fabricated `event_id`;
- current runtime date used as `event_time`;
- asset aliases such as `AI芯片` or `GPU` instead of `国产 AI 芯片`;
- missing entities such as `AI 芯片`, `GPU`, or `EDA`;
- extra explanatory text if the provider ignores JSON-only instructions.

## Compliance

This system is only for event research and market analysis. It does not output buy, sell, target price, guaranteed return, leverage advice, or automatic trading instructions.
